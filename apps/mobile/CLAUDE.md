## `expo prebuild --clean` wipes the Share Extension Xcode target

`ios/` and `android/` are gitignored and regenerated by Expo's CNG
(Continuous Native Generation) — `expo prebuild --clean` deletes and
recreates them from scratch. Two files under `ios/ShareExtension/` are
force-committed to git despite that (`Info.plist`,
`ShareViewController.swift`) because they're the source for a custom
native Share Extension target that Expo has no plugin to manage —
prebuild does not know this target exists.

Running `--clean` (or any prebuild that recreates `ios/` from nothing)
**deletes the Share Extension target from the Xcode project entirely**
and drops the `com.apple.security.application-groups` entitlement from
the main app (that one is now declared in `app.json`'s
`ios.entitlements` and will survive future prebuilds — this is the only
part that self-heals).

After any clean prebuild, before the share-to-app feature works again:

1. `git checkout -- apps/mobile/ios/ShareExtension/` if the two source
   files were also wiped from disk (they will be, but git still has
   them since they're tracked).
2. In Xcode: add a new **Share Extension** target, bundle ID
   `com.kulesza.carrot.ShareExtension`.
3. Point the new target's source at the existing
   `ios/ShareExtension/ShareViewController.swift` and `Info.plist`
   instead of the boilerplate Xcode generates — don't let it overwrite
   them.
4. Add the **App Groups** capability to the new target, with
   `group.com.kulesza.carrot` (must match `SHARE_APP_GROUP` in
   `src/utils/pendingShare.ts` and `appGroupId` in
   `ShareViewController.swift` — same ID hardcoded in both places).
5. Embed the extension in the main app (Xcode does this automatically
   when the extension target is added with the main app as the host).
