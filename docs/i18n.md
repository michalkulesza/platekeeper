# i18n Implementation Plan

## Decisions (resolved 2026-06-09)

| Area | Decision |
|---|---|
| Library | `react-i18next` + `i18next` + `i18next-browser-languagedetector` |
| Languages | `en` (English), `de` (Deutsch), `pl` (Polski), `fr` (Français), `es` (Español) |
| Translation files | `apps/web/src/locales/{en,de,pl,fr,es}.json` — one file per language, single namespace, nested by feature |
| Persistence | DB (`user_preferences.language`) = source of truth; `localStorage` = fast local cache |
| Initial load | Read `localStorage` first for instant render; confirm/override from DB after auth |
| Language switcher (authed) | HeroUI `Select` on Settings page, language names shown in native script |
| Language switcher (unauthed) | Globe icon (top-right corner) on Login + Register pages, `localStorage` only |
| API errors | Left untranslated in this pass |
| Pluralization | i18next plural keys (`_one` / `_few` / `_many` / `_other`) — required for Polish |
| Date formatting | Replace hardcoded `"en-US"` with active locale via a `useLocale()` hook |

## Implementation steps

### 1. Install packages
```
pnpm --filter web add react-i18next i18next i18next-browser-languagedetector
```

### 2. Backend — add `language` column
- Add `language: Mapped[str]` column to `UserPreferences` SQLAlchemy model (default `"en"`, `VARCHAR(10)`)
- Add `language` field to `UserPreferencesOut` and `UserPreferencesUpdate` Pydantic models
- Create Alembic migration

### 3. Create translation JSON files
`apps/web/src/locales/en.json` is the source of truth. Structure nested by feature:
```json
{
  "common": { "save": "Save", "cancel": "Cancel", ... },
  "nav": { "recipes": "Recipes", "mealPlan": "Meal plan", ... },
  "auth": { "signIn": "Sign in", "email": "Email", ... },
  "recipes": { "addRecipe": "Add recipe", ... },
  "settings": { "language": "Language", "weekStart": "Week starts on", ... },
  "allergens": { "gluten": "Gluten-containing cereals", ... },
  "timers": { "timerDone_one": "Timer done", "timerDone_other": "{{count}} timers done", ... }
}
```
Translate all keys into `de.json`, `pl.json`, `fr.json`, `es.json`.

### 4. i18next config + provider
Create `apps/web/src/i18n.ts` — initialise i18next with `LanguageDetector`, import all locale JSONs statically (bundle is small, no lazy loading needed).

Wrap app in `I18nextProvider` in `main.tsx` (outermost provider, before `BrowserRouter`).

### 5. Auth sync — wire DB preference to i18next
In `AuthContext`, after fetching user preferences:
- Call `i18n.changeLanguage(preferences.language)`
- Write to `localStorage` (handled automatically by `LanguageDetector`)

On language change in Settings:
- Call `i18n.changeLanguage(lang)`
- PATCH `/preferences` with `{ language: lang }`

### 6. Replace hardcoded strings
Replace all JSX string literals with `t('key')` calls using the `useTranslation()` hook. Key files:
- `App.tsx` — timer modals + plural strings
- `Sidebar.tsx` — nav labels
- `LoginPage.tsx`, `RegisterPage.tsx` — form labels, buttons
- `SettingsPage.tsx` — allergens, intolerances, preferences
- `RecipesPage.tsx` + `RecipeDetailModal.tsx` — action menus
- `MealPlanPage.tsx` — page headers
- `ShoppingListPage.tsx` — page headers
- `BellPopover.tsx` + `NotificationHistoryContext.tsx` — notification text
- `TimerContext.tsx` — service worker notification strings (best-effort; SW runs outside React)

### 7. Fix hardcoded locale in date formatting
- `RecipesTable.tsx:59` — replace `"en-US"` with `i18n.language`
- `MealPlanPage.tsx:117,159` — replace `"en-US"` with `i18n.language`
- Expose via a `useLocale()` hook: `const { locale } = useLocale()` → returns current i18next language

### 8. Language switcher UI
**Settings page** — add a HeroUI `Select` in the account/preferences section:
```
Language: [English ▾]
```
Options: `en → English`, `de → Deutsch`, `pl → Polski`, `fr → Français`, `es → Español`

**Login + Register pages** — add a small globe icon button (top-right, absolute positioned) that opens a dropdown with the same 5 options. Updates `localStorage` only (no auth yet).

## Out of scope (this pass)
- Backend API error message translation
- RTL language support
- Per-route lazy loading of translation files
- Translating service worker push notification body strings
